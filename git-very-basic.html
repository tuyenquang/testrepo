Git very basic

INTRODUCTION

What’s the problem?

  • As teams develop software, it is common for multiple stages of the same software to be deployed in different sites.

  • Bugs or features of the software are often only present in certain stages.

  • Therefore, for the purposes of locating and fixing bugs, it is vitally important to be able to retrieve and run different stages of the software to determine in which stage the problem occurs

What’s git?

Git is a free and open source distributed version control system (VCS) designed to handle everything from small to very large projects with speed and efficiency.

What’s version control system (VCS)? Version control systems are software tools that help software teams manage changes to source code over time. As development environments have accelerated, version control systems help software teams work faster and smarter

Why should you use git as a developer?

Feature branches provide an isolated environment for every change to your codebase.

How does git record changes overtime?

  • Every time you commit (or save the state of your project), Git basically takes a picture of what all your files look like at that moment.

  • Git thinks about its data more like a stream of snapshots.

WORKING WITH GIT

  • With Git, your’s files can have three states:

    • Modified files have chaged but have not committed it to your database yet.

    • Staged your files have marked a modified files for the next commit snapshot.

    • Committed means that the data is safely stored in your local database.

  • Respectively, you need to remember four areas:

    • Workspace area (working directory) area where you work

    • Staging area save current staged files

    • Local repository save commited files

    • Remote repository a copy of local repository saving in cloud (like github, gitlab, bitbucket…)

A SUCCESSFUL GIT BRANCHING MODEL

In 2010, Vincent Driessen recommended A successful Git branching model

Branches:

  • develop branch serves as the branch for ongoing development. All feature branches are merged into this branch. Your team should always keep the develop branch stable.

  • feature branch is used to develop new features or enhancements. They branch off from the develop branch and are merged back when the feature is complete.

  • master/main branch represents the production-ready code. It should only contain stable, bug-free code.

  • release branch is used for preparing a new production release. Once all features for a release have been added, this branch is merged into both master and develop branches.

  • hotfix branch is created to quickly fix critical bugs found in the production code. They branch off from the main branch and are merged back into both master and develop branches.
